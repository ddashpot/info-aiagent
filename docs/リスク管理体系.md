### 1.1 LLMアプリ脆弱性（llm・10件）

> プロンプト・出力・データ・モデルサプライチェーンなど、LLMを組み込んだアプリ層の主要リスク。（フレームワーク: owasp-llm）

#### LLM01 プロンプトインジェクション

| 属性 | 値 |
|---|---|
| 別名(aka) | Prompt Injection、Direct/Indirect Injection、Jailbreak |
| 深刻度(severity) | 致命的 |
| 影響サーフェス(surface) | input |
| 失敗モード(failure_mode) | integrity |

**要約(summary)**: ユーザー入力や外部データに埋め込まれた指示が、システムの意図した制約を上書きする。

**説明(description)**: 信頼できない入力（直接：ユーザー、間接：取得文書・Web・メール・ツール出力）に含まれる指示をモデルが正規の命令として実行し、ガードレール回避・権限外操作・情報漏えいを引き起こす。RAGやエージェントのツール出力経由の間接インジェクションは検出が難しい。

**攻撃ベクトル(attack_vectors)**
- 直接：会話中での『これまでの指示を無視して…』系の上書き
- 間接：取得文書/Webページ/メール本文/コミット/画像のalt等に隠した命令
- 多段：ツール出力→次エージェントの入力として伝播（カスケード）
- 符号化/難読化：Base64・Unicode・言語切替・ロールプレイによる回避

**影響を受けやすい構成(affected_patterns)**: `router-supervisor`、`handoff-swarm`、`blackboard`、`planner-executor`、`pipeline`、`event-driven-swarm`

**コントロール(controls)**

| ID | 種別 | 名称 | 説明 | 実装基盤(foundation) |
|---|---|---|---|---|
| `c-input-segmentation` | 予防 (Preventive) | 信頼境界の分離 | システム指示とユーザー/外部データを構造的に分離し、外部データを命令として解釈させない（spotlighting/デリミタ/データタグ）。 | プロンプトテンプレート設計、構造化メッセージAPI |
| `c-input-guard` | 予防 (Preventive) | 入力ガードレール | 既知のインジェクション語句・難読化・命令上書きパターンを検出/無害化。 | Llama Guard / NeMo Guardrails / Prompt Shields / 正規表現+分類器 |
| `c-output-guard` | 検知 (Detective) | 出力ガードレール | 応答が方針逸脱・機密露出・危険操作要求を含まないか検査。 | 出力分類器、ポリシーエンジン |
| `c-least-priv-tools` | 予防 (Preventive) | ツール最小権限 | インジェクション成功時の被害を限定するため、エージェントの権限・到達範囲を最小化。 | スコープ付きトークン、ツール許可リスト |
| `c-hitl-highrisk` | 対応 (Responsive) | 高リスク操作の人手承認 | 送信・決済・削除・権限変更など不可逆操作はHuman-in-the-loop。 | 承認ワークフロー、監査ログ |

**フレームワーク対応**
- MITRE ATLAS: AML.T0051 LLM Prompt Injection、AML.T0051.000 Direct、AML.T0051.001 Indirect、AML.T0054 LLM Jailbreak
- NIST AI RMF: MEASURE 2.7、MANAGE 2.2、MANAGE 4.1

**検査プローブ(probes)**: `pi_direct_override`、`pi_roleplay`、`pi_obfuscation`、`pi_indirect_doc`、`pi_ignore_previous`

**検知シグナル(signals)**
- system指示の開示
- 拒否ポリシーの逸脱
- 禁止トピックへの応答
- 外部指示への追従

**参考(references)**
- https://genai.owasp.org/llmrisk/llm01-prompt-injection/

#### LLM05 不適切な出力処理

| 属性 | 値 |
|---|---|
| 別名(aka) | Improper Output Handling、Insecure Output Handling |
| 深刻度(severity) | 高 |
| 影響サーフェス(surface) | output |
| 失敗モード(failure_mode) | integrity |

**要約(summary)**: モデル出力を検証せず下流へ渡し、XSS/SQLi/SSRF/コマンド実行等を誘発する。

**説明(description)**: LLM出力をHTML/SQL/シェル/URL/コードとして無検証で消費すると、生成された悪性ペイロードが下流システムで実行される。エージェントのツール引数生成で特に危険。

**攻撃ベクトル(attack_vectors)**
- 生成HTML→XSS
- 生成SQL→SQLi
- 生成コマンド→RCE
- 生成URL→SSRF
- Markdown画像によるデータ外送

**影響を受けやすい構成(affected_patterns)**: `pipeline`、`agents-as-tools`、`planner-executor`、`dag-workflow`

**コントロール(controls)**

| ID | 種別 | 名称 | 説明 | 実装基盤(foundation) |
|---|---|---|---|---|
| `c-output-encoding` | 予防 (Preventive) | 出力エンコード/サニタイズ | 下流コンテキストに応じてエスケープ/サニタイズ。LLM出力は信頼しない。 | コンテキスト別エスケープ、許可リスト |
| `c-param-tools` | 予防 (Preventive) | パラメータ化ツール | 自由文ではなく型付き・スキーマ検証済み引数のみ実行。 | JSONスキーマ検証、型付きツールIF |
| `c-egress-allowlist` | 予防 (Preventive) | 下流到達先の制限 | ツールが叩けるURL/ホスト/コマンドを許可リスト化（SSRF対策）。 | Egress制御、ネットワークポリシー |

**フレームワーク対応**
- MITRE ATLAS: AML.T0053 LLM Plugin Compromise
- NIST AI RMF: MEASURE 2.6、MANAGE 4.1

**検査プローブ(probes)**: `output_xss_payload`、`output_sqli_payload`、`output_markdown_exfil`

**検知シグナル(signals)**
- 生成された<script>
- 生成SQL/シェル
- 外部URLへの画像/リンク埋め込み

**参考(references)**
- https://genai.owasp.org/llmrisk/llm05-improper-output-handling/

#### LLM06 過剰なエージェンシー

| 属性 | 値 |
|---|---|
| 別名(aka) | Excessive Agency |
| 深刻度(severity) | 高 |
| 影響サーフェス(surface) | tools |
| 失敗モード(failure_mode) | authorization |

**要約(summary)**: 過剰な機能・権限・自律性により、誤動作や悪用時の被害が拡大する。

**説明(description)**: 不要なツール・広すぎる権限・人手確認なしの自律実行により、誤判断やインジェクション成功時に重大操作（送金/削除/送信）が実行される。エージェントの中核リスク。

**攻撃ベクトル(attack_vectors)**
- 不要ツールの誘発呼び出し
- 広域権限の悪用
- 確認なし不可逆操作
- 権限の連鎖的拡大

**影響を受けやすい構成(affected_patterns)**: `agents-as-tools`、`planner-executor`、`manager-worker`、`hierarchical-supervisor`、`market-bidding`

**コントロール(controls)**

| ID | 種別 | 名称 | 説明 | 実装基盤(foundation) |

## 4. 方法論（methodology）

- **note**: 各テストは『入力生成戦略 × 判定オラクル』の組で表される。軸内は相互排他、組合せで網羅(CE)。未知脆弱性は、生成戦略で仕様の境界を押し、オラクルで仕様逸脱を判定して炙り出す。
- **generation_strategies**:
  - id=seed / name=種コーパス / desc=既知プローブの基準ペイロード(OWASP/ATLAS対応)。
  - id=mutation / name=変異 / desc=符号化・言い換え・言語切替・分割・注入ラッパで種を系統変換。
  - id=grammar / name=文法/テンプレート / desc=テンプレートや文法から構造的にペイロードを生成（拡張点）。
  - id=search / name=探索(回帰的深掘り) / desc=弱兆候を起点に成功要因を強める方向へビーム探索的に追撃。
  - id=stateful / name=多ターン/状態 / desc=会話・記憶をまたぐ多段で永続化・累積効果を誘発。
- **oracles**:
  - id=signature / name=シグネチャ一致 / desc=内容パターン照合（秘密/カナリア/反射/追従語）。 / detectors=["compliance", "secret_pattern", "canary_leak", "system_prompt_echo", "injection_echo", "regex"]
  - id=invariant / name=不変条件(ポリシー)違反 / desc=『拒否を維持』『境界を越えない』等の不変条件の違反。 / detectors=["refusal_absent"]
  - id=differential / name=差分/メタモルフィック / desc=安全/危険に変形した同義入力ペアの応答差で脆弱点を特定（拡張点）。 / detectors=["differential"]
  - id=anomaly / name=統計的異常 / desc=平常値からの逸脱（長さ・拒否率・文字種・エントロピー）。 / detectors=["anomaly"]
  - id=side_effect / name=副作用/ツール痕跡 / desc=ツール呼び出し・外部到達など観測可能な副作用。 / detectors=["tool_abuse", "url_exfil"]
  - id=judge / name=人手/LLM判定 / desc=自動判定が困難な意味的逸脱を人手またはLLMで判定（拡張点・受動プローブ）。 / detectors=[]
- **rationale**: 既知シグネチャだけでは新規・組み合わせ・コンテキスト依存の脆弱性を見逃す。挙動ベースの差分・変異・回帰探索で『仕様から逸脱する応答』を異常として検出し、人手トリアージに回す。
- **triage**: 異常検出は確証ではない。各フラグに再現手順・入出力・該当する可能性のあるカタログIDを付与し、重大度推定とともにレポート化して人手トリアージ・登録（既知化）へ回す。回帰スイートに取り込むことで継続的に再検査する。
